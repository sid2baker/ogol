# Sequence DSL Specification

## 1. Purpose

This specification defines a higher-level orchestration DSL for Ogol.

It sits above:

- the **Machine DSL**, which defines what one machine can do and what it reports
- the **Topology DSL**, which defines composition, supervision, observations, and restart policy

The Sequence DSL is the canonical authoring and model-definition layer for
application orchestration in Ogol.

Its job is to answer:

- what should the application do next?
- in what order?
- under which conditions?
- with what timeout, retry, and failure behavior?

A machine describes capabilities and current condition. A sequence describes how
to use those capabilities to implement application Auto behavior.

---

## 2. Architectural Position

Ogol should be layered like this:

### 2.1 Machine layer

The machine layer defines one machine boundary.

A machine owns:

- public skills
- public signals
- public status
- local internal behavior
- local hardware integration

This layer answers:

- what can this machine do?
- what is its current condition?

### 2.2 Topology layer

The topology layer composes machines into one node-level runtime.

A topology owns:

- deployment
- supervision
- dependency wiring
- observations
- restart policy
- failure propagation policy

This layer answers:

- which machines exist together?
- how do they observe each other?
- what should happen when one fails?

### 2.3 Sequence layer

The Sequence DSL consumes machine contracts and topology-visible state.

It does not define:

- raw hardware behavior
- raw machine internals
- restart trees
- supervisor semantics

This layer answers:

- how should the application behave in Auto?
- what ordered orchestration flow should be executed?
- when should progress wait, branch, retry, or fail?

---

## 3. Core Position

The Sequence DSL is the preferred source of truth for application orchestration.

The current Ogol Machine and Topology runtime is the primary execution backend
for artifacts generated from Sequence DSL definitions.

So the architectural position is:

> Sequence DSL -> validated canonical sequence model -> generated runtime artifacts -> current Ogol Machine/Topology backend

IEC 61499 remains an alignment target for the backend architecture, not the
primary framing of the source language.

---

## 4. Implementation Model

The Sequence DSL is intended to be implemented with Elixir Spark.

The normative source of truth is the Spark-defined Sequence DSL together with
its validation and the validated canonical sequence model derived from it.

The runtime does not execute surface syntax directly. It executes generated
artifacts derived from that validated model.

The expected compilation pipeline is:

1. author writes Sequence DSL using Spark
2. Spark builds declarative sections and entities
3. machine and topology references resolve
4. validations run
5. the validated canonical sequence model is produced
6. orchestration runtime artifacts are generated
7. HMI metadata is generated
8. diagnostics and trace metadata are generated

This means examples in this spec are illustrative. The normative model is the
Spark DSL structure and its validated canonical representation, not any one
pseudo-textual surface notation.

---

## 5. Design Principles

The Sequence DSL shall follow these principles:

1. **Machine contracts first**
   Sequences orchestrate exposed skills and observed public state. They do not
   reach into machine internals.

2. **Topology owns fault tolerance**
   Restart policy, dependency loss handling, and supervision stay in topology,
   not in sequence source.

3. **Operator-readable ordering**
   The source form should read like a sequence sheet:
   wait, do, confirm, continue, recover, fail.

4. **Deterministic execution**
   Order, blocking, timeout, branching, and failure behavior must be explicit.

5. **Safe-by-construction bias**
   Preconditions, invariants, waits, timeouts, and bounded recovery are
   first-class.

6. **Single generated model**
   Runtime, autogenerated HMI, curated HMI, simulation, diagnostics, and docs
   derive from the same validated canonical sequence model.

7. **Backend portability**
   The source should lower cleanly into the current Ogol event/state runtime and
   remain conceptually aligned with IEC 61499-style execution.

---

## 6. Scope

The Sequence DSL is intended for:

- Auto mode application logic
- machine-to-machine sequencing
- startup and shutdown sequences
- homing and preparation flows
- guarded skill orchestration
- bounded retry and recovery
- cycle execution for cells, stations, and small lines

It is not intended to be:

- a general-purpose programming language
- a hardware driver language
- a raw machine capability definition language
- a UI layout language
- a deployment language

---

## 7. Machine Contract Assumptions

A sequence may only depend on machine contracts that are explicitly exposed.

The minimum machine contract is:

- **skills**
  invokable public requests/events
- **status**
  readable public machine state, facts, outputs, and health
- **signals**
  observable public notifications

The Sequence DSL should primarily use:

- skill invocation
- status observation

The Sequence DSL should not depend on:

- private machine memory
- raw hardware endpoints
- hidden mutable cross-machine state
- private implementation states unless intentionally projected as public status

---

## 8. Topology Contract Assumptions

Topology remains responsible for supervision and resilience.

Topology concerns include:

- one-for-one restart policy
- safe-stop policy on dependency loss
- escalation rules
- degraded operation policy
- dependency observation routing
- runtime ownership and activation

Sequence logic may observe topology-visible state, but topology alone defines:

- restart trees
- supervision structure
- dependency restart policy
- escalation policy

Sequence failure is not the same thing as topology restart behavior.

---

## 9. Conceptual Model

The Sequence DSL is built on six concepts:

1. **Observed public state**
   facts, status, and topology-visible state projected from machine and
   topology contracts

2. **Skills**
   invokable machine capabilities

3. **Steps**
   ordered units of orchestration behavior

4. **Guards**
   conditions required for legal progress

5. **Invariants**
   conditions that must remain true while the sequence is active

6. **Failures and recovery**
   explicit bounded handling for timeout, refusal, or loss of progress

---

## 10. Typed References

Surface syntax may be concise, but internal references must be typed and
resolved.

The minimal reference classes are:

- `SkillRef`
- `StatusRef`
- `SignalRef`
- `TopologyRef` when topology-visible state is referenced

Examples:

- `clamp.close` lowers to `SkillRef(machine: :clamp, skill: :close)`
- `robot.status.homed` lowers to `StatusRef(machine: :robot, item: :homed)`
- `conveyor.signal.started` lowers to `SignalRef(machine: :conveyor, item: :started)`

The dotted form is author-facing sugar. The typed reference is the semantic
form.

---

## 11. Expression Scope

Expressions may combine:

- literals and constants
- boolean and comparison operators
- `StatusRef` values
- `TopologyRef` values
- derived predicates built from those references

Expressions must not directly read:

- private machine memory
- raw hardware endpoints
- hidden shared mutable state

Signal references are not ordinary durable expressions. They must be used only
through explicit signal-oriented constructs, since they are edge-triggered and
not level semantics by default.

Local aliases for repeated predicates or references are intentionally out of
scope for v1. A future version may add named aliases or derived predicates, but
the first version should keep expression scope explicit and direct.

---

## 12. Illustrative Source Shape

Illustrative surface notation may look like:

```text
sequence Auto {
  invariant not estop
  invariant clamp.status.air_ok

  proc startup {
    wait clamp.status.open timeout 2s else fail "clamp not open"
    wait robot.status.homed timeout 10s else fail "robot not homed"
  }

  proc cycle {
    wait feeder.status.part_present
    do clamp.close
    wait clamp.status.closed timeout 2s else fail "clamp failed"
    do robot.pick
    wait robot.status.at_pick timeout 5s else fail "robot not at pick"
  }

  run startup

  repeat {
    run cycle
  }
}
```

This notation is schematic, not normative. The real implementation target is a
Spark-defined DSL that produces the same canonical sequence model.

`proc` is the local reusable block construct inside a Sequence definition. It
does not name the DSL layer itself.

A `sequence` defines one orchestration unit. Top-level statements inside the
sequence body form that sequence's root entrypoint, while nested `proc` blocks
define reusable sub-sequences that may be invoked with `run`.

---

## 13. Spark-Shaped Structural Example

One possible Spark-flavored authoring shape would be:

```elixir
sequence :auto do
  invariant expr(not(estop()))
  invariant expr(ref(:clamp, :status, :air_ok))

  proc :startup do
    wait ref(:clamp, :status, :open), timeout: 2_000, else: fail("clamp not open")
    wait ref(:robot, :status, :homed), timeout: 10_000, else: fail("robot not homed")
  end

  proc :cycle do
    wait ref(:feeder, :status, :part_present)
    do_skill :clamp, :close
    wait ref(:clamp, :status, :closed), timeout: 2_000, else: fail("clamp failed")
    do_skill :robot, :pick
    wait ref(:robot, :status, :at_pick), timeout: 5_000, else: fail("robot not at pick")
  end

  run :startup

  repeat do
    run :cycle
  end
end
```

This example is illustrative too. The important point is the structure:

- typed references
- explicit validation targets
- a clean path to generated artifacts

## 14. Statements

The minimal Sequence DSL should support:

### 14.1 `do`

Invoke a machine skill.

```text
do clamp.close
do robot.pick
```

### 14.2 `wait`

Block until a machine or topology condition becomes true.

```text
wait clamp.status.closed
wait robot.status.at_pick timeout 5s else fail "robot stalled"
```

Loss of a required referenced machine or topology-visible dependency during an
active `wait` causes sequence failure unless topology policy intercepts that
outcome and replaces it with broader stop, restart, or escalation behavior.

### 14.3 `if`

Conditionally execute a branch.

```text
if inspection.status.failed {
  do reject_gate.reject
} else {
  do conveyor.advance
}
```

### 14.4 `run`

Invoke a named sub-sequence.

```text
run startup
run cycle
```

### 14.5 `repeat`

Repeat a block indefinitely.

```text
repeat {
  run cycle
}
```

### 14.6 `fail`

Enter an explicit orchestration failure.

```text
fail "clamp failed"
```

### 14.7 `when`

Attach a guard to one statement.

```text
do robot.pick when clamp.status.closed
```

`when` is an execution-time precondition guard. If the guard is false when the
statement is reached, the guarded statement fails immediately unless it is
wrapped in an explicit conditional or future handling construct. `when` does
not implicitly skip or block.

---

## 15. Skill Invocation Outcomes

`do machine.skill` must have explicit invocation semantics.

The minimal invocation outcomes are:

- `accepted`
- `rejected`
- `errored`
- `request_timed_out`

There are two distinct timeout classes:

- **request boundary timeout**
  the skill invocation itself did not reach a definite accepted/rejected/error
  outcome in time
- **effect timeout**
  the later confirmation step, usually expressed with `wait`, did not observe
  the required durable effect in time

The rules are:

- `do` succeeds only if invocation is accepted
- `rejected` is an immediate sequence failure unless a future handling
  construct exists
- `errored` is an immediate sequence failure unless explicitly handled
- effect confirmation is separate and should normally be expressed with `wait`

So:

- `do` means "request the action"
- `wait` means "confirm the state change"

This preserves the distinction between accepted intent and confirmed effect.

---

## 16. Status vs Signal Semantics

This distinction is normative.

### 16.1 Status

Status observations are:

- level-triggered
- durable
- replayable
- preferred for orchestration correctness

### 16.2 Signals

Signal observations are:

- edge-triggered
- potentially ephemeral
- not inherently replayable

Signal-based orchestration should only be used when explicitly intended.

Signals must not be silently treated as durable state.

So the rule is:

- prefer waits over status
- use waits over signals only when edge semantics are actually required

Sequence correctness depends on machine contracts exposing durable public status
sufficient to distinguish acceptance, progress, completion, readiness for the
next step, and failure where needed.

---

## 17. Invariants

Invariants express conditions that must remain true while the sequence is
active.

Examples:

```text
invariant not estop
invariant door_closed
invariant pressure_ok
```

Sequence invariants express conditions under which the sequence may continue.
Topology determines the broader system consequence of invariant violation.

---

## 18. Recovery

Recovery must remain explicit and bounded.

Example:

```text
wait clamp.status.closed timeout 2s else {
  do clamp.open
  do clamp.close
  wait clamp.status.closed timeout 2s else fail "clamp failed twice"
}
```

The Sequence DSL should not hide retries.

Topology-level restart remains separate. Sequence recovery is about bounded
logical recovery, not process supervision.

---

## 19. Validation

Validation is a first-class part of the language design.

A conforming implementation should validate at least:

1. every referenced machine exists in the topology
2. every referenced skill exists and is public
3. every referenced status item exists and is public
4. every referenced signal exists and is public if signal waits are used
5. signal-based waits are explicitly declared as signal waits
6. `wait` conditions are boolean
7. invariants are boolean
8. every timeout literal is valid
9. every timeout branch is structurally valid
10. every referenced sub-sequence exists
11. the sequence call graph is valid
12. every recovery path is bounded
13. causes of failure are preserved through lowering

Recommended additional checks:

- referenced skills that are never available in legal states
- wait conditions that can never become true
- contradictory invariants
- repeated bodies that cannot make progress
- failure branches that silently discard root cause

Where possible, invalid references should fail at compile time.

---

## 20. Canonical Sequence Model

The Sequence DSL should normalize into a validated canonical sequence model
built from:

- typed references
- sequence definitions
- steps
- guards
- timeouts
- failure branches
- invariants
- runtime projection metadata

Illustrative forms:

```text
SkillRef {
  machine: :clamp
  skill: :close
}
```

```text
StatusRef {
  machine: :clamp
  item: :closed
}
```

```text
SignalRef {
  machine: :clamp
  item: :cycle_started
}
```

```text
Step {
  kind: Do
  target: SkillRef(:clamp, :close)
}
```

```text
Step {
  kind: Wait
  condition: StatusRef(:clamp, :closed)
  timeout: 2s
  on_timeout: Fail("clamp failed")
}
```

This model is the basis for:

- runtime generation
- autogenerated HMI generation
- diagnostics generation
- trace metadata generation
- documentation generation

---

## 21. Generated Artifacts

A Sequence DSL definition should generate:

- the validated canonical sequence model
- orchestration controller definition(s)
- HMI metadata
- diagnostics schema
- trace/event schema
- documentation metadata

This generated artifact set is how the DSL becomes operational inside Ogol.

---

## 22. Sequence Instance Runtime Model

At runtime, a sequence executes as a **sequence instance**.

A sequence instance should minimally include:

- current sequence definition
- current active step
- call stack for nested `run`
- active wait condition
- active timeout
- current blocked reason
- active invariants
- failure state
- trace context

Recommended lifecycle states:

- `idle`
- `starting`
- `running`
- `waiting`
- `failed`
- `completed`
- `stopping`
- `aborted`

This runtime model is the basis for sequence HMI projection and diagnostics.

---

## 23. Lowering to Current Ogol Runtime

The current Ogol Machine and Topology implementation should be treated as the
execution backend for the Sequence DSL.

That means:

- **machine modules** remain the capability/state boundary
- **topology modules** remain the supervision and observation boundary
- **generated orchestration controller machine(s)** execute sequence logic

Expected lowering:

1. machine contracts resolve from the declared topology
2. sequence definitions normalize into the canonical sequence model
3. the compiler generates one or more controller machines
4. controller machines invoke machine skills using existing machine composition primitives
5. topology observations project dependency state and signals into controller events
6. controller state machines execute the sequence deterministically

Lowering must preserve diagnostic causality across:

- sequence source
- the canonical sequence model
- generated runtime artifacts
- runtime failure projection and HMI/diagnostic output

So the current backend already provides the right execution shape:

- event/state execution
- guarded transitions
- explicit timeouts
- explicit signals
- topology observation routing
- supervision owned by topology

This is the primary backend story. IEC 61499 alignment remains a backend
architecture note, not the primary source-language framing.

---

## 24. HMI Derivation

HMI should be derived from the validated canonical sequence model plus runtime
state.

At minimum, a sequence HMI should expose:

- current sequence
- current step
- current active wait
- current active machine skill
- active invariants
- blocked reason
- timeout/recovery/failure state
- the machine statuses currently being waited on

The architecture should distinguish:

- **machine HMI**
  local machine state and capabilities
- **sequence HMI**
  orchestration progress and blocked reason
- **optional autogenerated operator/debug HMI**
  a generic operational surface derived from the same model

---

## 25. IEC 61499 Alignment

The Sequence DSL is not raw IEC 61499 syntax.

Instead, it should compile into a backend that is IEC 61499-aligned or
equivalent in event/state architecture.

Recommended conceptual mapping:

- sequence steps -> controller ECC states and transitions
- `do` -> event emission / skill invocation
- `wait` -> guarded waiting transitions
- invariants -> supervisory fault transitions or monitors
- failures -> explicit fault states and diagnostics
- machine boundaries -> function-block-like capability units
- topology composition -> composite function-block-like orchestration structure

The important rule is:

- author in Sequence DSL
- execute in generated event/state artifacts

## 26. Example

```text
sequence Auto {
  invariant not estop
  invariant clamp.status.air_ok

  proc startup {
    wait clamp.status.open timeout 2s else fail "clamp not open"
    wait robot.status.homed timeout 10s else fail "robot not homed"
  }

  proc cycle {
    wait feeder.status.part_present

    do clamp.close
    wait clamp.status.closed timeout 2s else fail "clamp failed"

    do robot.pick
    wait robot.status.at_pick timeout 5s else fail "robot not at pick"

    do robot.place
    wait robot.status.at_place timeout 5s else fail "robot not at place"

    do clamp.open
    wait clamp.status.open timeout 2s else fail "clamp did not open"
  }

  run startup

  repeat {
    run cycle
  }
}
```

This should lower into:

- a validated canonical sequence model
- one or more orchestration controller machines
- machine skill invocations
- topology observation wiring
- current Ogol supervision/runtime behavior

---

## 27. Non-Goals for Version 1

Version 1 excludes:

- arbitrary user-defined functions
- unrestricted mutable variables
- general concurrency syntax
- raw hardware addressing in sequence source
- direct access to machine internals from sequences
- topology supervision definitions in sequence source
- hidden shared mutable state across machines
- custom UI/widget layout syntax
- deployment/target syntax
- arbitrary foreign code in sequence source

These may be added later only if they preserve deterministic and readable
control behavior.

---

## 28. Conformance Summary

An implementation conforms to this specification if it:

- provides a sequence-layer source language above machines and topology
- treats machine contracts as the public execution boundary
- leaves supervision and restart policy in topology
- uses Spark or an equivalent declarative DSL system to produce a validated
  canonical sequence model
- lowers that model into generated runtime artifacts over the current Ogol
  Machine/Topology backend or an equivalent event/state backend
- exposes sequence runtime state for HMI derivation and diagnostics

## 30. Preferred Position

The preferred Ogol architecture is:

- **Machine DSL**
  local capability and state definition
- **Topology DSL**
  composition, supervision, observations, and restart strategy
- **Sequence DSL**
  application Auto logic over exposed machine contracts

This keeps each layer honest:

- machines describe what they can do
- topology describes how they live together safely
- sequences describe how the application uses them

That is the recommended long-term direction for Ogol.
