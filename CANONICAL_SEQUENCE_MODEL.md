# Canonical Sequence Model Specification

## 1. Purpose

This document defines the validated canonical model produced from
[SEQUENCE_DSL.md](SEQUENCE_DSL.md).

Its job is to answer:

- what data structure is the real semantic output of Sequence DSL authoring?
- what must be preserved through validation and lowering?
- what runtime and HMI projection data must be available?

This model sits between:

- the Spark-defined Sequence DSL source
- generated runtime artifacts over the current Ogol Machine/Topology backend

It is the primary semantic contract for:

- lowering
- diagnostics
- trace identity
- HMI projection
- documentation and tooling

---

## 2. Architectural Position

The architecture is:

> Sequence DSL source -> validated canonical sequence model -> generated runtime artifacts -> current Ogol Machine/Topology runtime

The canonical sequence model is not a parser AST and not a runtime snapshot.

It is:

- validated
- resolved
- typed
- stable enough for tooling and diagnostics

It must preserve the authored sequence semantics without requiring runtime code
to reinterpret source syntax.

---

## 3. Normative Commitments

A conforming implementation of the canonical sequence model shall satisfy these
rules:

1. Every machine, skill, status, signal, and topology reference is typed and
   resolved.
2. Every executable step has a stable identity.
3. Every `run` edge can be reconstructed as a nested execution frame.
4. Every `wait` has an explicit condition shape.
5. Every failure path preserves diagnostic cause.
6. Every generated runtime artifact can be traced back to canonical step
   identity.
7. The model exposes enough runtime projection metadata for autogenerated HMI,
   traces, and debugging.

---

## 4. Model Overview

The canonical sequence model is built from:

- sequence definitions
- procedure definitions
- typed references
- expressions
- executable steps
- guard conditions
- timeout semantics
- recovery and failure paths
- runtime projection metadata

The top-level model shape is conceptually:

```text
CanonicalSequenceModel {
  version: 1
  sequence: SequenceDefinition
  procedures: [ProcedureDefinition]
  refs: [TypedRef]
  diagnostics: DiagnosticsMetadata
  projection: ProjectionMetadata
}
```

The exact storage format is implementation-defined. The semantic fields are
normative.

---

## 5. Identity Model

Stable identity matters for:

- trace events
- HMI step highlighting
- diagnostics
- runtime causality
- documentation

So the canonical model must assign stable identities to at least:

- sequence definitions
- procedure definitions
- executable steps
- timeout branches
- explicit failure sites

Illustrative shape:

```text
SequenceId = "auto"
ProcedureId = "auto.startup"
StepId = "auto.cycle.step.close_clamp"
```

The specific encoding is implementation-defined, but identities must be:

- deterministic for the same authored source
- stable across recompilation when the logical step remains the same
- available in runtime diagnostics and trace output

Source location metadata may supplement identity, but it must not replace it.

---

## 6. Typed References

The canonical model does not retain unresolved dotted strings as semantics.

The minimum typed reference classes are:

- `SkillRef`
- `StatusRef`
- `SignalRef`
- `TopologyRef`

### 6.1 `SkillRef`

```text
SkillRef {
  machine: :clamp
  skill: :close
}
```

Represents invocation of one public machine skill.

### 6.2 `StatusRef`

```text
StatusRef {
  machine: :robot
  item: :homed
}
```

Represents one durable public machine status item.

### 6.3 `SignalRef`

```text
SignalRef {
  machine: :conveyor
  item: :started
}
```

Represents one public machine signal. Signal refs are only legal in
signal-explicit constructs because they are edge semantics, not durable level
semantics.

### 6.4 `TopologyRef`

```text
TopologyRef {
  scope: :system
  item: :estop
}
```

Represents topology-visible or node-visible public state.

This is the preferred resolution form for examples like `estop`. A small set of
global orchestration-safe refs may appear unqualified in source, but they should
normalize into typed `TopologyRef` values in the canonical model.

---

## 7. Expression Model

Expressions in the canonical model are typed and resolved.

They may combine:

- literals
- boolean operators
- comparison operators
- `StatusRef`
- `TopologyRef`
- derived predicates built from those components

Signals are not ordinary expression operands unless the construct explicitly
admits signal semantics.

Illustrative shapes:

```text
Expr.And(
  Expr.Not(TopologyRef(scope: :system, item: :estop)),
  StatusRef(machine: :clamp, item: :air_ok)
)
```

```text
Expr.Equal(
  StatusRef(machine: :robot, item: :mode),
  Literal(:homed)
)
```

The canonical model should preserve enough structure for:

- validation
- HMI explanation
- diagnostics
- guard evaluation lowering

It must not collapse expressions into opaque strings.

---

## 8. Sequence and Procedure Definitions

The canonical model distinguishes:

- one root sequence definition
- zero or more reusable procedures

Illustrative shapes:

```text
SequenceDefinition {
  id: SequenceId
  name: :auto
  root: StepBlockId
  invariants: [InvariantId]
  procedures: [ProcedureId]
}
```

```text
ProcedureDefinition {
  id: ProcedureId
  name: :startup
  body: StepBlockId
}
```

The sequence root body is the top-level entrypoint. `run` creates nested
execution into reusable procedures and returns to the caller when the invoked
procedure frame completes successfully.

---

## 9. Step Model

Every executable unit in the canonical model is a step with:

- stable `id`
- `kind`
- optional `guard`
- optional `timeout`
- optional failure or recovery branch
- projection metadata

Illustrative base shape:

```text
Step {
  id: StepId
  kind: StepKind
  guard: Expr | nil
  timeout: TimeoutSpec | nil
  on_timeout: Branch | nil
  projection: StepProjection
}
```

### 9.1 Core Step Kinds

The minimum canonical step kinds are:

- `DoSkill`
- `WaitStatus`
- `WaitSignal`
- `If`
- `RunProcedure`
- `Repeat`
- `Fail`
- `Block`

#### `DoSkill`

```text
DoSkill {
  target: SkillRef
}
```

Represents one skill invocation.

#### `WaitStatus`

```text
WaitStatus {
  condition: Expr
}
```

Represents a wait over durable level semantics.

#### `WaitSignal`

```text
WaitSignal {
  signal: SignalRef
}
```

Represents an explicitly edge-triggered wait.

#### `If`

```text
If {
  condition: Expr
  then_branch: StepBlockId
  else_branch: StepBlockId | nil
}
```

#### `RunProcedure`

```text
RunProcedure {
  procedure: ProcedureId
}
```

Represents nested control transfer into a named procedure.

#### `Repeat`

```text
Repeat {
  body: StepBlockId
}
```

#### `Fail`

```text
Fail {
  message: "clamp failed"
}
```

#### `Block`

```text
Block {
  steps: [StepId]
}
```

Represents ordered step composition.

---

## 10. Guard Semantics

`when` does not create a new execution mode. It lowers into an execution-time
guard on the affected step.

Illustrative shape:

```text
GuardedStep {
  step: DoSkill(target: SkillRef(machine: :robot, skill: :pick))
  guard: StatusRef(machine: :clamp, item: :closed)
}
```

If the guard is false when the step is reached:

- the step fails immediately
- no implicit blocking occurs
- no implicit skip occurs

unless the authored sequence wrapped that step in an explicit conditional or
future handling construct.

---

## 11. Wait and Timeout Model

The canonical model must distinguish at least:

- request boundary timeout
- effect confirmation timeout

### 11.1 Request Boundary Timeout

This applies to the invocation boundary of `DoSkill`.

Illustrative shape:

```text
InvocationPolicy {
  timeout: 1_000
}
```

It covers failure to obtain a definite invocation outcome such as `accepted`,
`rejected`, or `errored`.

### 11.2 Effect Confirmation Timeout

This applies to `wait`-style progress confirmation.

Illustrative shape:

```text
TimeoutSpec {
  kind: :effect
  duration_ms: 5_000
}
```

It covers failure to observe the required durable condition or explicit signal
within the expected time.

### 11.3 Dependency Loss During Wait

If a required referenced machine or topology-visible dependency disappears while
an active wait is pending, the canonical outcome is sequence failure.

Topology may intercept that outcome and replace it with broader stop, restart,
or escalation behavior, but the sequence model itself should preserve the failed
wait as the direct local cause.

---

## 12. Invocation Outcome Model

The canonical model must represent the minimum invocation outcomes defined by
[SEQUENCE_DSL.md](SEQUENCE_DSL.md):

- `accepted`
- `rejected`
- `errored`
- `request_timed_out`

Illustrative shape:

```text
InvocationOutcome =
  :accepted
  | :rejected
  | :errored
  | :request_timed_out
```

Only `accepted` allows forward progress from the `DoSkill` step itself.

Effect confirmation remains a separate concern and should be modeled by later
wait-oriented steps unless a richer machine skill contract eventually says
otherwise.

---

## 13. Failure Cause Model

Diagnostic causality is a normative requirement.

The canonical model must preserve failure cause in a way that survives:

- source validation
- normalization
- lowering
- runtime execution
- HMI/trace projection

Illustrative shape:

```text
FailureCause {
  kind: :timeout
  site: StepId
  ref: StatusRef(machine: :clamp, item: :closed)
  message: "clamp failed"
  parent: FailureCause | nil
}
```

Typical cause kinds include:

- `:rejected`
- `:errored`
- `:request_timed_out`
- `:effect_timed_out`
- `:guard_failed`
- `:invariant_failed`
- `:dependency_unavailable`
- `:explicit_fail`

Failure branches may enrich cause metadata, but they must not discard the
originating cause.

---

## 14. Runtime Projection Metadata

The canonical model must carry enough metadata to support generated HMI and
diagnostic output.

At minimum it should support projection of:

- current sequence
- current procedure frame
- current active step
- current active wait
- current timeout
- current blocked reason
- current invariants
- current failure cause

Illustrative shape:

```text
StepProjection {
  label: "Wait for clamp to close"
  summary: "waiting for clamp.status.closed"
  refs: [StatusRef(machine: :clamp, item: :closed)]
}
```

The exact wording is not normative, but the model must retain enough structure
for explanation and tooling.

---

## 15. Sequence Instance Model

The canonical model must support generation of a runtime **sequence instance**
that contains at least:

- sequence definition id
- current procedure frame
- active step id
- call stack of nested `run` frames
- active wait
- active timeout
- blocked reason
- active invariants
- failure cause
- trace context

Illustrative shape:

```text
SequenceInstance {
  sequence: SequenceId
  frames: [Frame]
  active_step: StepId
  state: :running | :waiting | :failed | :completed
  failure: FailureCause | nil
}
```

And:

```text
Frame {
  procedure: ProcedureId | :root
  block: StepBlockId
  index: non_neg_integer()
}
```

This is the minimum runtime projection shape needed for sequence HMI and
diagnostics.

---

## 16. Lowering Expectations

Lowering from the canonical sequence model into generated controller artifacts
must preserve:

- step identity
- typed reference identity
- nested frame structure
- timeout class
- failure cause
- runtime projection metadata needed for HMI and traces

The generated runtime does not need to store the canonical model verbatim, but
it must preserve these semantics observably.

---

## 17. Example

Illustrative normalized fragments:

```text
SequenceDefinition {
  id: "auto"
  name: :auto
  root: "auto.root"
  procedures: ["auto.startup", "auto.cycle"]
}
```

```text
ProcedureDefinition {
  id: "auto.cycle"
  name: :cycle
  body: "auto.cycle.body"
}
```

```text
Step {
  id: "auto.cycle.step.close_clamp"
  kind: DoSkill(target: SkillRef(machine: :clamp, skill: :close))
  guard: nil
  timeout: nil
  on_timeout: nil
}
```

```text
Step {
  id: "auto.cycle.step.wait_clamp_closed"
  kind: WaitStatus(condition: StatusRef(machine: :clamp, item: :closed))
  timeout: TimeoutSpec(kind: :effect, duration_ms: 2_000)
  on_timeout: Fail(message: "clamp failed")
}
```

---

## 18. Conformance Summary

An implementation conforms to this specification if it:

- produces a typed, validated canonical sequence model from Sequence DSL source
- assigns stable identities to sequence steps and procedures
- distinguishes status waits from signal waits
- distinguishes request-boundary timeout from effect timeout
- preserves failure cause through lowering
- exposes enough runtime projection metadata for HMI, traces, and diagnostics
